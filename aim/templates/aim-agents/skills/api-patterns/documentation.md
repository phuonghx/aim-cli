# Documenting an API

> Docs are the onboarding ramp. Clear ones turn into adoption.

## What an OpenAPI spec should cover

```
Spell out:
├── Every route, with a worked example
├── The shape of each request and response
├── How callers authenticate
├── The format errors come back in
└── Throttling limits and headers
```

## Marks of a reference people actually use

```
Round it out with:
├── A getting-started path that yields a first success fast
├── A focused guide to authentication
├── Exhaustive endpoint reference
├── Guidance on handling failures
├── Runnable snippets in more than one language
└── A changelog so consumers can track breakage
```
